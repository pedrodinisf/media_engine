OVERVIEW 001 · FRAMEPULSE

Typed media pipelines that survive every interface.

framepulse is a universal media-processing engine. Hand it a video, audio file, image, URL, document, or live stream — pull back typed artifacts (transcripts, frame analyses, structured extractions, embeddings, summaries, search results) through one cache-coherent substrate. One engine, five transports, thirty-four capability-named operations, thirty-plus pluggable backends.

QUICK START 002 · ONBOARDING

From uv sync to a typed transcript in under a minute.

Four stages: install, ingest, run, inspect. Every command goes through the same engine the REST API and the MCP server use, so an artifact created on the shell shows up everywhere immediately.

1

Install · macOS

Homebrew · uv · ~2 min

Three Homebrew packages then uv sync. The optional extras (api, mcp, transcribe-mlx, embed, vlm-cloud, postgres, …) are pure opt-in — install only the ones you'll actually call.

bash · macOS · Apple Silicon or Intel
# 1. Prerequisites — ffmpeg is required by every media op
brew install uv ffmpeg

# 2. Clone and sync — minute one
git clone https://github.com/<you>/framepulse.git
cd framepulse
uv sync --extra api --extra mcp --extra transcribe-mlx --extra embed

# 3. Sanity check — should print 31 capability-named ops
uv run med ops
uv run med config

For Apple Silicon: --extra transcribe-mlx pulls in mlx-whisper (uses the Neural Engine). For Intel or cross-platform, swap in --extra vlm-cloud to route through Gemini/Claude instead.

2

Ingest a real video

yt-dlp · streams sha256 · ffprobe-classifies

Hand framepulse any URL yt-dlp can resolve (YouTube, Vimeo, direct HLS, m3u8 playlist, raw .mp4) or a local file. The engine streams the bytes through sha256 as they land, calls ffprobe to classify the kind (Video / Audio / Image), and lands the file in the sha256-sharded permanent store. Stdout is just the id.

bash · first real download
# A short public-domain test clip (Wikimedia · Big Buck Bunny, ~10 MB)
URL="https://upload.wikimedia.org/wikipedia/commons/transcoded/c/c0/Big_Buck_Bunny_4K.webm/Big_Buck_Bunny_4K.webm.480p.vp9.webm"

VID=$(uv run med acquire-url "$URL")
echo "acquired: $VID"          # sha256 hex, 64 chars

# Inspect what just landed
uv run med show $VID            # typed Video metadata
uv run med ls --kind Video      # registry view

# Any URL yt-dlp can handle works the same way:
#   uv run med acquire-url "https://www.youtube.com/watch?v=<id>"
#   uv run med acquire-url "https://vimeo.com/<id>"
# Local files take the same shape:
#   uv run med acquire ~/Downloads/clip.mp4
3

Run the pipeline

cost preview → confirm → dispatch

Generic runner. Cost preview prints first — --yes skips the prompt, --dry-run exits after the estimate without spending. Pass an explicit backend with --backend, or rely on the op's selection logic (model prefix → backend).

bash · Video → Audio → Transcript → Analysis
# Video → Audio (ffmpeg, no cloud cost)
AUDIO=$(uv run med extract-audio $VID)

# Audio → Transcript — local on Apple Silicon via mlx-whisper
TX=$(uv run med run audio.transcribe \
    --input $AUDIO --backend mlx-whisper --yes)

uv run med show $TX              # segments, language, model used

# Transcript → Analysis — pick any backend (mlx-lm / claude / gemini)
uv run med run intelligence.summarize \
    --input $TX \
    --param focus="key arguments" \
    --backend mlx-lm --yes
4

Inspect

cache · lineage · ledger · search

Every artifact is content-addressed, typed, and queryable. Walk the provenance tree, browse the spend ledger, hybrid-search the corpus.

bash · introspection
# browse the typed registry
uv run med ls --kind Transcript --limit 10
uv run med ls --kind Analysis   --limit 10

# upstream provenance — depth-limited, cycle-safe tree
uv run med lineage $TX --depth 6

# actual spend (per-execution rows; cache hits never billed)
uv run med cost summary
uv run med cost ls --op intelligence.summarize

# hybrid search — semantic + fulltext via Reciprocal Rank Fusion
uv run med search "main argument of the talk" --mode hybrid --top-k 5
5

Prove the cache

reanalysis at zero upstream cost

The killer feature: change one knob, only that op re-runs. Everything upstream returns from cache instantly. Both summaries persist side-by-side; both are reachable by id.

bash · reanalysis without re-spending
# Try a second analytical lens on the same transcript
# — only the summarize op re-runs; transcribe + extract are cache hits
uv run med run intelligence.summarize \
    --input $TX \
    --param focus="counterarguments and open questions" \
    --backend mlx-lm --yes

# Both summaries now live in the catalog
uv run med ls --kind Analysis --limit 5

# Re-running the exact same call is FREE — instant cache hit
uv run med run intelligence.summarize \
    --input $TX \
    --param focus="key arguments" \
    --backend mlx-lm --yes

CHARTER 003 · WHAT & WHY

One substrate, many surfaces.

What it is

A universal substrate for typed media processing.

Hand framepulse any media artifact — a video, an audio file, an image, a URL, a PDF, a live HLS stream — and pull back typed outputs through one cache-coherent engine. Thirty-one capability-named operations apply uniformly across every input: ingest, extract, transcribe, diarize, analyze, summarize, embed, search, render. Fourteen typed artifact kinds flow between them.

What makes it universal is the strict separation between what and how. Operations are named for the capability — audio.transcribe, never mlx_whisper.transcribe — and the technology lives in a swappable backend layer behind every op. mlx-whisper, pyannote, Gemini, Claude, sentence-transformers, pgvector, OpenCLIP, RapidOCR… the registry has thirty of them across local and cloud. Same op, different provider, no collision.

The cache is content-addressed at sha256 depth. Every artifact id is the hash of (kind, op, op_version, backend, backend_version, params, sorted_input_ids). Identical inputs and params yield identical ids — deterministically, cross-machine, cross-transport. Re-running anything is free. Bumping op.version or swapping a backend is the only way to invalidate.

Pipelines are data, not code. A YAML profile describes a DAG of op invocations. The engine compiles it, runs the graph through an async TaskGroup executor with resource-bounded parallelism, retries cloud-flaky backends, emits a structured event stream, and writes a spend-ledger row for every actual execution.

Why it exists

To stop reinventing the same plumbing.

Every team that touches media ends up writing the same five hundred lines: an ffmpeg subprocess wrapper, a Whisper invoker, a cache scheme that breaks on the second model swap, retry logic for the cloud VLM, a homegrown cost log. Each iteration is subtly incompatible with the next. Each new pipeline is a new program.

framepulse factors all of that into one substrate. New media tools become profiles (YAML data), not new programs (code). The same op catalog answers from a shell, an HTTP request, an MCP tool call from an AI agent, or a Python import. The same cache row is reachable from each of them. The same cost ledger captures every cent, regardless of which transport drove it.

Reanalysis is a one-knob change. Tweak the focus param on a summarize call — only that node re-runs. The transcribe, the extract, the embed, the chunk: all return from cache instantly. Try a different prompt, a different model, a different backend — the cache keys are versioned, so the new shape lives alongside the old without collision.

  • Any media · 14 typed artifact kinds, content-addressed
  • Any backend · 30 registered, local + cloud, hot-swappable
  • Any transport · CLI, daemon, REST, MCP, Python
  • Any deployment · laptop, Docker, Kubernetes, Terraform
TOPOLOGY 004 · ARCHITECTURE

Five layers, four transports, one engine.

Every transport hits the same Engine. The engine resolves an op + backend, checks the content-addressed cache, and either returns the cached artifact or runs the op — persisting the result, indexing it, ledgering the cost. Pipelines are the same machinery fanned out across an async DAG.

PRINCIPLES 005 · DESIGN INVARIANTS

Ten engine invariants worth knowing.

These aren't aspirational. Each one is enforced by the type system, the registry, the cache schema, the test suite, or some combination. Bending one breaks the cache-coherence guarantees that make the rest fast.

01Capability-named operations
audio.transcribe, never mlx_whisper.transcribe. Technology lives in the backend layer. Names validated at registration.
02Content-addressed cache
Every artifact id is sha256(kind, op, op_v, backend, backend_v, params, sorted(inputs)). Identical inputs ⇒ identical id ⇒ cache hit.
03Swappable backends
Same op, different provider. Backend name + version are in the cache key, so mlx-whisper and Gemini transcripts coexist without collision.
04Profiles are data
YAML DAG or Markdown-with-frontmatter, no Python required. med profile run my-pipeline --input source=$ID.
05Zero domain opinions
No sentiment dimensions, no speaker schemas in the core. Profile-supplied JSON schemas validate via a built-in zero-dep validator.
06Streaming-first events
Every op emits OpStarted / Progress / ArtifactReady / OpCompleted / OpFailed. Daemon and REST forward over their own transports.
07MCP-native
Every registered op auto-exports as an MCP tool. Allow-list defaults to read-only search.*; writeful ops opt in.
08DAG, not linear pipeline
Fan-out / fan-in with resource-bounded parallelism. Per-node retry. Partial completion: a failed node fails dependents; siblings keep running.
09Cost-aware
op.cost_estimate() everywhere; --dry-run previews the DAG total; med cost summary shows actuals from the post-hoc ledger.
10Resource-aware
Declared resources ⇒ asyncio.Semaphore. 1 VLM at a time on Apple Silicon. Tunable per-deployment via resources.yaml.

CLI — ad-hoc work, scripts, CI

The most direct surface. med auto-detects a running daemon (50 ms ping) and routes through it; falls back to in-process otherwise. Use this for one-shot tasks, shell-friendly scripting, CI jobs, and quick reprocessing.

  • Ingest: med acquire, med acquire-url, med acquire-live
  • Generic runner: med run <op> --input ID --param K=V --backend B — cost preview, --yes to skip prompt, --dry-run to estimate only
  • Read: med ls, med show, med lineage, med ops, med config
  • Batch: med batch <urls.txt> --concurrency 5 — fan an op over a list through the DAG executor
  • Search: med search "query" --mode hybrid --top-k 5
  • Spend: med cost summary · med cost ls --since 2026-05-01 --op intelligence.extract
bash · full pipeline in shell
VIDEO=$(med acquire ~/Downloads/talk.mp4)
AUDIO=$(med extract-audio $VIDEO)
TX=$(med run audio.transcribe --input $AUDIO --yes)
med run intelligence.summarize --input $TX \
    --param focus="main argument" --yes

med lineage $TX --depth 6
med cost summary --since 2026-05-22

Profiles — reusable pipelines as YAML

When a workflow stops being ad-hoc, write a profile. Pipeline profiles describe a YAML DAG of op invocations; prompt profiles are Markdown with frontmatter, shorthand for a single VLM/LLM call. Both are validated at load — missing ops, unknown backends, and cycles all fail before the first task runs.

Discovery searches ~/.config/media_engine/profiles/, the repo's bundled profiles/ dir, and any --profile-dir override. Later directories override earlier ones.

yaml · profiles/my-pipeline.yaml
name: my-pipeline
kind: pipeline
inputs:
  - { name: source, kind: video }
graph:
  - { id: audio,      op: video.extract_audio,       inputs: { in: source } }
  - { id: transcript, op: audio.transcribe_diarized, inputs: { audio: audio } }
  - { id: analysis,   op: intelligence.analyze,
      inputs: { t: transcript },
      params: { schema: schemas/my_extract.json } }
outputs: [analysis]
bash · run a profile
med profile ls
med profile show my-pipeline
med profile run my-pipeline --input source=$VIDEO_ID

Daemon — warm models for fast dev loops

med daemon start keeps a long-lived Engine.open_session() alive across CLI invocations. Warm Whisper, hot Hugging Face cache, shared resource semaphores. Every med command auto-routes through it via a 50 ms ping; falls back to in-process otherwise. No code change required on the caller side.

  • Lifecycle: med daemon start · med daemon stop · med daemon status · med daemon logs --tail 200
  • Watch events: med events tail --follow · med events history --since 2026-05-22
  • Transport: Unix socket at ~/.config/media_engine/daemon.sock, JSON-line frames, protocol-versioned with backward-compat check on connect
  • Periodic GC: daemon sweeps orphan workdirs older than the retention window (default 24 h) on a configurable interval (MEDIA_ENGINE_GC_INTERVAL)
bash · daemon-backed batch loop
med daemon start
med events tail --follow &     # live event stream in another pane

for v in ~/talks/*.mp4; do
  med profile run my-pipeline --input source=$(med acquire "$v")
done

med daemon stop

REST API — service-style integration

med api start boots FastAPI under uvicorn. Async-first: POST /run and POST /pipelines return a job_id immediately; subscribe to GET /jobs/{id}/events for SSE progress, or poll GET /jobs/{id} for status.

Bearer-token auth with sha256-hashed storage; raw secret shown once at creation. OpenAPI at /openapi.json. Probes at /health + /ready are unauthenticated for kubelet. Orphan job recovery on lifespan boot flips frozen running rows to failed after a crash.

bash · REST round-trip
med api start --host 0.0.0.0 --port 8000 &
TOKEN=$(med api token create)   # secret printed once to stdout

JOB=$(curl -s -X POST http://localhost:8000/run \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"op":"acquire.upload","params":{"source_path":"/data/x.mp4"}}' \
    | jq -r .job_id)

curl -N -H "Authorization: Bearer $TOKEN" \
     http://localhost:8000/jobs/$JOB/events   # SSE stream

Web UI — local-first browser surface

A SvelteKit single-page app is bundled inside the engine container at /ui and served by the same FastAPI process that powers the REST API. Full GUI parity with the CLI: anything you can do with med <verb> you can do from the browser. The path of least resistance for users who don't want a shell — while the CLI stays first-class for power users + CI.

Panels shipped (commits 39–50 + 3 audits + 2 docs syncs): Ingest (upload / URL probe / livestream / batch), Run (op picker + schema-driven form + live cost preview), Jobs (live SSE-driven table + per-job detail with events / op runs / outputs / failure envelope + cancel), Catalog (paginated list + kind chip filter + per-kind preview affordances for all 14 artifact kinds), Catalog detail with Svelte Flow + dagre lineage graph viewer, Search (sync fulltext / semantic / hybrid with type-as-you-go feedback), Cost (per-{op,backend,namespace} rollup bars + monthly burn projection + paginated drill-down ledger), Profile workspace (split visual DAG composer + CodeMirror YAML editor + 650 ms live compile + fork-this examples library), Settings (six-tab panel: Tokens · Backends · Plugins Extras · Plugins Catalog · Storage · Config). Bundled with the +ui multi-stage Dockerfile (Node-free runtime image) and six regenerable screenshots in docs/web_ui/.

Bootstrap: mint a bearer token via the CLI, then paste it into the browser at /ui/setup. Zero new endpoints, zero chicken-and-egg. Persists to localStorage (or sessionStorage via the session-only toggle).

Security: CSP scoped to /ui/* (default-src 'self' with 'wasm-unsafe-eval' for pdf.js + 'unsafe-inline' styles for Svelte's scoped CSS); same-origin only by default (MEDIA_ENGINE_CORS_ORIGINS opens dev origins); SSE routes accept ?token=… since EventSource can't set headers (loopback-target, v1.x nonce hardening path queued).

# One-time: build the SPA dist tree
pnpm -C web install --frozen-lockfile

# Mint a token (talks to the cache, no running API needed)
med api token create --label web-ui

# Boot uvicorn + mount /ui + (optionally) open the browser
med web start --open

Wheel installs from PyPI ship the prebuilt dist tree via hatch force-include — pip install media_engine[api] gets the UI for free with no Node toolchain on the host. See docs/web_ui.md for the panel-by-panel tour.

MCP stdio — Claude Code & agent integration

Every registered op auto-exports as an MCP tool: group.verb → group__verb, with the params model's JSON Schema, an injected input_artifact_ids array, and a backend enum. Artifacts surface as media://<kind>/<id> resources via resources/list and resources/read.

Security posture: default allow-list is read-only (search.semantic, search.fulltext, search.hybrid). Writeful ops opt in explicitly via --allow; --deny always wins on conflict.

bash · register with Claude Code
# default: read-only (search.* only)
claude mcp add framepulse "med mcp serve"

# expose specific writeful ops
claude mcp add framepulse \
    "med mcp serve --allow audio.transcribe --allow intelligence.summarize"

# or open everything (trusted local-only setups)
claude mcp add framepulse "med mcp serve --allow '*'"

Python API — embed in another tool

The public surface is small and stable: Engine, Pipeline, Artifact, Kind, register_op, register_backend. Two factories: Engine.open_quick() for one-shot scripts, Engine.open_session() for long-lived processes (warm models, shared semaphores, periodic GC).

python · embedded use
import asyncio
from media_engine import Engine
from media_engine.bootstrap import register_all

register_all()  # populate op + backend registries

async def go():
    with Engine.open_session() as e:
        [video] = await e.run(
            "acquire.upload", source_path="/data/talk.mp4",
        )
        [audio] = await e.run("video.extract_audio", inputs=[video.id])
        [tx]    = await e.run("audio.transcribe", inputs=[audio.id])
        print(tx.metadata["text"][:200])

asyncio.run(go())

PROFILES 009 · DATA, NOT CODE

Reusable pipelines as YAML.

Engine principle 4: profiles are data, not code. A profile is a named, parameterized pipeline stored as YAML (for explicit DAGs) or Markdown with frontmatter (for prompt-driven shorthand). The engine compiles it to a runtime Pipeline and dispatches through the same DAG executor that powers Engine.run — caching, lineage, semaphores, retry, partial completion, all of it.

FLAVOR 01 · PIPELINE

YAML DAG — explicit multi-op

Used when a workflow chains 3+ ops, needs fan-out / fan-in, or wants explicit input wiring. The graph field is a list of op invocations; the engine compiles it to a runtime.dag.Pipeline via topological wave sort, refusing to start on cycles, unresolved refs, or duplicate node ids.

yaml · profiles/analysis-full/profile.yaml
profile_schema_version: "1.0"
name: analysis-full
kind: pipeline
description: Spoken video → speaker-diarized intelligence + Markdown report

inputs:
  - { name: source, kind: video }

graph:
  - id: audio
    op: video.extract_audio
    inputs: { in: source }
  - id: transcript
    op: audio.transcribe_diarized
    inputs: { audio: audio }
    params: { language: en }
  - id: analysis
    op: intelligence.analyze
    inputs: { t: transcript }
    params: { prompt_path: analysis-full/prompt.md }
  - id: report
    op: report.session
    inputs: { a: analysis }

outputs: [transcript, analysis, report]
FLAVOR 02 · PROMPT

Markdown + frontmatter — single VLM/LLM call

Used when a workflow is one VLM call wrapped around a custom system prompt + (optionally) a JSON output schema. The Markdown body becomes the system_prompt param to default_op. Five starter prompt profiles ship under profiles/*.md (video-knowledge, technical-academic, diy-electronics, cooking-recipes, general-custom).

markdown · profiles/technical-academic.md
---
profile_schema_version: "1.0"

- Supporting evidence
- Open questions

Field reference & semantics.

profile_schema_version Always "1.0" for now. Lets the loader migrate without breaking older files. Required.
name Used by med profile run NAME and POST /pipelines { profile_name }. Must be unique across all discovery directories. Server-side kebab-case validation ([a-z0-9-]+) on POST /profiles blocks path traversal.
kind pipeline for YAML DAGs, prompt for Markdown shorthand. The discriminator that picks the loader path.
inputs[].name + kind Caller-supplied source artifacts. name is the label referenced from graph[].inputs; kind is the lowercase Kind value (video, audio, image, …) checked at compile time. med profile run NAME --input name=ID binds them.
graph[].id / op / inputs / params / backend / depends_on One entry per node. id is unique within the profile; op is a registered op name (med ops to list); inputs maps op-input labels to source names or upstream node ids (list form [source] also accepted); params is op-specific and validated against the op's params_model; backend pins a specific backend; depends_on declares extra dependencies that cascade failure.
outputs Node ids whose artifacts are returned. Defaults to the leaf nodes if omitted.
Discovery order Three directories, later wins: (1) ~/.config/media_engine/profiles/ (your personal profiles, editable from the Web UI), (2) <engine repo>/profiles/ (bundled starters, read-only in the Web UI), (3) any --profile-dir override. med profile ls + GET /profiles show the resolved set.
Run-time semantics Compiled pipelines dispatch through runtime.dag.Pipeline: ready nodes run concurrently in an asyncio.TaskGroup; per-resource semaphores (apple_neural_engine: 1, cloud_concurrent: 4, …) bound parallelism; every node's output is cached by (op, version, backend, params, input_ids); per-node retry policy applies (cloud defaults to 3 attempts exponential); a failed node fails its dependents but independent siblings keep running. Result is partial completion, surfaced through the Web UI Jobs detail page.
Fork-and-edit is the supported edit loop. The Web UI profiles index shows a 🔒 badge on bundled profiles + a one-click fork-this modal. Pick a kebab-case name; the workspace opens with the renamed copy under {config_dir}/profiles/. From there: edit the YAML (650 ms live compile catches mistakes), Save (POST /profiles), Run (Sources modal → POST /pipelines with inline pipeline_yaml so unsaved drafts execute). The CLI loop is the same: cp profiles/X.yaml ~/.config/media_engine/profiles/X-mine.yaml → edit → med profile run X-mine.

Full field-by-field tutorial in docs/writing_a_profile.md; the analysis-full reference pipeline is documented in docs/profile_analysis_full.md.

COST MODEL 010 · MONEY

Two stages, both honest.

The engine never hides spend. You see what it will probably cost before you press go, and what it actually cost after. Cache hits and thin composite wrappers are explicitly not double-billed.

STAGE 01 · UPFRONT

Cost estimate

Every backend implements cost_estimate(inputs, params) → CostEstimate{local_seconds, cloud_cents, tokens_in, tokens_out}. The engine sums them across a DAG before a single op runs.

med run prints the preview and asks before spending. --yes skips the prompt; the global --dry-run prints and exits without spending.

Pricing tables in backends/_pricing.py follow the tiered ≤200K vs >200K input-token pattern for Gemini, Claude, OpenAI, and Voyage. Local backends report zero cloud_cents.

STAGE 02 · POST-HOC

Actual spend ledger

Every real op execution writes a row to cost_log with backend-reported usage (cents, input tokens, output tokens, duration). Cache hits and records_cost=False composites are not billed.

med cost summary rolls up per-op spend. med cost ls --since 2026-05-01 --op intelligence.extract shows the per-execution receipts.

The ledger is namespaced — multi-tenant deployments see only their own tenant's rows. Same data through REST when the cost endpoints land in Phase 5.

Reanalysis is free by construction. Re-running the same profile with one knob tweaked re-uses every upstream cache hit. Change intelligence.summarize's focus param from “main argument” to “counterarguments” — only the summarize node re-runs; ingest, audio extract, and transcribe all return instantly from cache. The new analysis lives next to the old one, both reachable by id.
