### F-001 [P1] ¬∑ `scripts/build_web.sh` exits non-zero on pnpm 11 ‚Äî **FIXED**

**File:** `scripts/build_web.sh:23`
**Problem:** `pnpm install --frozen-lockfile` exits 1 under pnpm ‚â• 11 with `ERR_PNPM_IGNORED_BUILDS` (esbuild's post-install lifecycle script is not in `onlyBuiltDependencies`). `set -euo pipefail` kills the script before reaching `pnpm build`. A fresh contributor following CLAUDE.md's documented onboarding hits this on first run.
**Fix:** Add `pnpm.onlyBuiltDependencies: ["esbuild"]` to `web/package.json`. (Alternative: pin `packageManager: "pnpm@10.x"`. Less future-proof.)
**Regression test:** `bash scripts/build_web.sh && test -f media_engine/web/dist/index.html` (already implied by CI, will pass once fixed).

### F-002 [open observation, possible latent bug] ¬∑ `verify_observability.sh` ffmpeg-lines spec flakes

**File:** `web/tests/e2e/flows/observability.spec.ts:111`
**Problem:** The "Phase 6.7: Logs tab has a source filter and surfaces ffmpeg lines" spec polls for `N lines` counter > 0 within a 10 s window. Behavior observed during this audit:
- **Step 0 baseline** (after web build): run 1 = 2/3 fail (under contention with parallel verify_settings), run 2 = 3/3 pass.
- **Step 6 tip-commit sweep** (after all 18 fixes landed): runs 1, 2, 3 = all 2/3 fail (one spec consistently times out, no contention).

**Direct LogLine pump inspection** outside Playwright confirms 41 `log_line` events ARE persisted in `cache.event_log` with the correct `job_id`. So the engine path is healthy. The failure has to be in the SSE-replay ‚Üí page-render race path. Hypotheses worth investigating:
1. Test-machine disk pressure (5.6 GB free / 98% full) slowing ffmpeg's stdout flush enough that LogLine events fire AFTER the 10 s budget.
2. The `consoleLines` capture is empty when the test fails ‚Äî suggesting the page never even logged its SSE handshake. Possibly a navigation race where the page mounts but the EventSource doesn't subscribe before `record_event` flushes.
3. Playwright accumulated browser state (test-results dir kept across reruns; cleared with no effect during this audit).

**Audit confirmation that this is NOT introduced by the audit's fixes:** `git diff main..HEAD -- media_engine/ops/video/extract_audio.py media_engine/runtime/log_pump.py media_engine/runtime/engine.py media_engine/api/sse.py media_engine/backends/sample_frames/ffmpeg_uniform.py` is empty.

**Recommendation:**
- Investigate as a Phase-7 / follow-up audit task. Likely a real flake ‚Äî possibly disk-pressure-induced; possibly a SSE handshake race.
- Short-term: bump the `.toPass({ timeout: 10_000, ... })` to 20-30 s and capture the SSE handshake state in the failure message.
- Don't ship as a "fix" in this audit since the root cause is unconfirmed and the workaround is a timeout bump.

### F-008 [P0, P5] ¬∑ `video.comprehend` doesn't forward `start_s`/`end_s` to `video.extract_audio` ‚Äî **FIXED**

**File:** `ops/video/comprehend.py:315-317`
```python
[audio] = await run_op("video.extract_audio", inputs=[video.id])
```
**Problem:** Bare call ‚Äî no params. The audio side is extracted from the FULL video while the transcript (via `audio.transcribe_diarized`, lines 319-333) and the frame sampling (via `video.sample_frames`, lines 345-354) both honor `start_s`/`end_s` correctly. Result: a 10 s window from a 60 min meeting transcribes 10 s of audio (correct) but extracts 60 min of audio (wrong ‚Üí cache miss ‚Üí wasted time ‚Üí silent timeline misalignment if any downstream consumer reads the audio metadata).

**Same class as commit 02fc1e9** (which forwarded start_s/end_s to `video.sample_frames`). The same fix was needed for `video.extract_audio` and got missed.

**Fix:** Build `ea_kwargs` paralleling the `td_kwargs` pattern at lines 319-330:
```python
ea_kwargs: dict[str, Any] = {}
if params.start_s is not None:
    ea_kwargs["start_s"] = params.start_s
if params.end_s is not None:
    ea_kwargs["end_s"] = params.end_s
[audio] = await run_op("video.extract_audio", inputs=[video.id], **ea_kwargs)
```
**Regression test:** `tests/test_op_video_comprehend.py` ‚Äî add a spec that invokes the composite with `start_s=2.0, end_s=5.0` and asserts the captured `ctx.run_op` call to `video.extract_audio` received those params. (Use the existing test scaffolding that mocks delegate calls.)

## Op-overlap analysis

Functional clusters worth calling out:

- **`audio.transcribe` + `audio.transcribe_diarized`**: clean. The composite is purely additive (transcribe + diarize + merge).
- **`intelligence.analyze` / `intelligence.classify` / `intelligence.summarize` / `intelligence.extract`**: extract is the leaf; the other three are thin composites over extract with different prompts/schemas. No duplication.
- **`search.semantic` / `search.fulltext` / `search.hybrid`**: standard RRF composite over two leaf search ops. Clean.
- **`video.multimodal` vs `video.comprehend`**: borderline overlap. Both accept a Video and produce structured analysis. `video.multimodal` is a single VLM call over the whole video (cloud-friendly, no diarization); `video.comprehend` fans out per-frame VLM calls + audio diarized transcript + SOTA-LLM synthesis. ## Design questions for the user

### DQ-1 ‚Äî Ephemeral marking on `audio.transcribe_diarized` (F-014)

Should the intermediate `Diarization` artifact created inside `audio.transcribe_diarized` be marked `metadata.ephemeral = True` (hidden from the catalog list)?

- **Arguments for ephemeral**: it's scaffolding for the merge; operators rarely care about the standalone diarization once it's merged into the transcript. Reduces catalog clutter.
- **Arguments against**: a standalone diarization (speaker timestamps without transcript text) is occasionally useful for analytics ‚Äî e.g. "who spoke when" without needing the words. Hiding it makes it harder to find.

**Recommendation**: keep visible by default; add a composite-level param `keep_intermediate_diarization: bool = True` so users can toggle.

### DQ-2 ‚Äî Ephemeral marking on `search.hybrid` sub-searches (F-015)

Same question for `search.hybrid`'s intermediate `search.semantic` + `search.fulltext` outputs.

**Recommendation**: mark as ephemeral. Unlike F-014, the sub-search outputs aren't useful standalone ‚Äî they're identical to running the sub-op directly, which the user can do explicitly. The composite producing two visible siblings every time is pure clutter.

### DQ-3 ‚Äî `video.multimodal` vs `video.comprehend` positioning

Both ops produce structured video analysis. Different cost profiles, different hardware requirements. Worth a doc page (`docs/video-ops.md` or a ¬ß15 in `architecture.md`) explaining when each is preferred. Currently undocumented.

### DQ-4 ‚Äî `list[str]` form widget gap (Web UI polish)

`web/src/lib/components/forms/SchemaForm.svelte` falls back to a text input for `list[str]` Params fields (currently used by `intelligence.classify.labels`, `intelligence.analyze.classify_labels`). Should a dedicated multi-input widget land in this audit (P2 polish) or wait for a Web-UI phase?

**Recommendation**: defer to a follow-up Web-UI polish phase; the text fallback works (comma-separated). Not blocking.
